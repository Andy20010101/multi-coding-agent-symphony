## Implementation

Workbench now exposes `MainVerificationGateRegistration` on the Verification path after the main verification evidence draft is complete. The model is anchored to the active goal/task, readiness state, evidence draft, latest controlled verification operation, command results, and existing goal event refs.

The registration panel renders only when these explicit inputs are present:

- readiness is true for the active task;
- `mainVerificationEvidenceDraft` is `draft-ready`;
- target main verification evidence ref is present;
- controlled verification run passed and exposes command results;
- no existing main verification ref is recorded for the task.

The rendered form uses the existing `event-plan-preview` and `event-plan-confirm` flow for `symphony goal gate`. `gate=main-verification` and `status=passed` are fixed fields. The operator still supplies the verifier id, runs dry-run preview first, and confirms only with the returned `planHash`. Confirm appends only the backend `main.verification-passed` event and refreshes goal-status, events, and next action.

The browser path does not run arbitrary shell commands, invoke models, write evidence files, read evidence bodies, merge, push, tag, publish, declare release readiness, or self-approve. A successful verification operation remains command evidence until this separate gate confirm is run.
