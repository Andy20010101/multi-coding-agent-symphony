v15 Workbench React/Vite Migration

Migrate the read-only Symphony Workbench display to React/Vite as a Chinese-first / 中文优先, low-density / 低信息密度, copy-only state surface while preserving v12 adoption safety, v14 Stage kernel behavior, ArtifactStore evidence boundaries, and Stage Charter HTML as a separate display artifact.

candidate v15-workbench-react-vite-migration JSON source of truth

Boundaries

api-contract-first

API contract before frontend migration

React/Vite work starts only after existing Workbench GET API shapes are inventoried and fixture-backed.

read-only-copy-only

Workbench remains read-only and copy-only

The browser UI may fetch local GET APIs and copy command text only. It must not expose browser controls for write, execute, retry, adopt, adopt-confirm, confirm-adoption, apply, rollback, delete, package install, dependency install, model invocation, real agent invocation, arbitrary file read, arbitrary path read, arbitrary path input, mutation trigger, or audit trigger actions.

symphony-artifactstore-boundary

.symphony stores summaries and refs only

.symphony continues to store only compact summary, ref, and pointer state such as active Stage pointers, latest run pointers, ids, statuses, timestamps, counts, artifact refs, registered local paths, top risk summaries, blocker summaries, next-action summaries, and copy-only command text. It must not store complete high-risk evidence, complete repair plans, complete blocked snapshots, or complete adoption-aware evidence. ArtifactStore remains canonical for TaskPackets, Harness output, evidence packages, full high-risk evidence, repair plans, blocked snapshots, adoption-aware evidence, patch artifacts, scaffold manifests, intake artifacts, and verifier-readable artifacts. React/Vite Workbench reads .symphony summaries/refs and /api/* aggregate results only; it must not treat .symphony as canonical evidence storage or bypass ArtifactStore to interpret high-risk evidence directly.

artifact-preview-contract

Artifact preview contract is explicit before React binding

Before React renders artifact previews, the API contract must expose stable artifact uri or ref, size or sizeBytes, mime, title or displayTitle, type, format, safeToRenderInline, truncated, sourceRunId, artifactKind, previewAvailable, and a consistent error envelope. React must not infer these fields from local paths or raw .symphony state.

stage-charter-html-separate

Stage Charter HTML is separate

Stage Charter HTML remains the JSON-derived Stage display artifact; the React/Vite Workbench is a local runtime UI and does not replace or edit it.

v13-v14-ui-principles

Preserve v13/v14 Workbench principles

The migrated Workbench stays Chinese-first / 中文优先, low information density / 低信息密度, read-only, copy-only, and state-focused. Visible labels, empty states, status text, command descriptions, and primary copy are Chinese unless stable JSON enums, ids, commands, or file paths must stay literal. The homepage defaults to Stage, goal, status, top risks, blocker, and next action only. Run, evidence, adoption, diagnostics, and raw JSON details stay folded or secondary. The migration is parity-first, not a UI redesign.

phased-fallback

Phased rollout with legacy fallback

Keep a safe legacy Workbench fallback until React bundle smoke and fixture parity gates pass. Missing bundles, build failures, or static asset serving failures must fall back to existing server-rendered or legacy static HTML behavior, and /api/* plus console snapshot JSON must remain v14-compatible. Fallback removal requires a later reviewer-approved removal step after parity passes.

kernel-preservation

Preserve v12/v14 kernels and gates

The migration must not modify the v12 adoption apply safety kernel, v12 fingerprint verification logic, v12 dirty worktree check logic, or v12 git apply --check logic. It must not rewrite the v14 Stage kernel, the Stage gate, blocked snapshot recovery, or ArtifactStore boundaries.

project-scope-non-goals

No GitHub/PR integration or mcas re-entry

The migration must not add GitHub / PR integration and must not make mcas the daily primary entry point again. React/Vite Workbench remains a local read-only status surface.

dependency-high-risk

React/Vite dependency introduction is high risk

React/Vite dependency introduction is an independent high-risk task. It must start plan-only, receive independent reviewer approval, use a separate commit or separate change group, never be auto-adopted, have no auto-adopt path, never be mixed with UI migration code in one large task, and require reviewer approval for package.json and pnpm-lock.yaml changes. After lockfile updates, run pnpm audit --audit-level high.

Top Risks

read-only-regression

Workbench accidentally gains browser write controls high

Keep server GET-only and require UI tests proving no write, execute, retry, adopt, adopt-confirm, confirm-adoption, apply, rollback, delete, package install, dependency install, model invocation, real agent invocation, arbitrary file read, arbitrary path read, arbitrary path input, mutation trigger, or audit trigger controls exist.

artifact-boundary-regression

Workbench bypasses ArtifactStore evidence boundaries high

Expose high-risk evidence through API summaries and ArtifactStore refs only; never treat .symphony as canonical evidence storage or parse high-risk evidence directly from .symphony.

artifact-contract-gap

Artifact preview contract omits React-safe rendering fields high

Require uri/ref, size or sizeBytes, mime, title/displayTitle, type, format, safeToRenderInline, truncated, sourceRunId, artifactKind, previewAvailable, and consistent error envelope fields before React preview binding.

api-drift

React components bind to unstable raw state fields medium

Introduce fixture-backed API contracts and a frontend API client boundary before component work.

dependency-supply-chain

React/Vite dependency introduction expands audit surface high

Treat dependency introduction as a separate high-risk plan-only task requiring independent review, a separate change group, no auto-adoption, package.json and pnpm-lock.yaml review, and pnpm audit --audit-level high after lockfile update.

charter-workbench-confusion

Stage Charter HTML is confused with the Workbench app medium

Document Stage Charter HTML as JSON-derived display-only output and keep Workbench runtime assets separate.

ui-parity-gap

Migration hides current Workbench diagnostics or recovery details medium

Use v14 fixtures for Stage, runs, adoption, blockers, artifacts, and readiness before replacing the UI.
