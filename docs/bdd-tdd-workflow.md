# BDD and TDD Workflow

This repository uses BDD and TDD as the default project creation method.

BDD = Behavior-Driven Development: behavior is written before implementation in a user-observable form.

TDD = Test-Driven Development: implementation starts from a failing automated test and proceeds through red, green, refactor.

## Behavior Layer

Behavior scenarios live in `features/` or in task artifacts generated by the orchestrator.

Preferred scenario shape:

```gherkin
Feature: Adapter conformance

  Scenario: A runtime adapter reports capabilities
    Given a registered runtime adapter
    When the orchestrator probes the adapter
    Then the capability report includes supported commands
    And the report includes session resume support
    And the report includes log collection strategy
```

Scenarios should describe observable behavior, not internal implementation details.

## Test Layer

Tests live in `tests/` until the implementation language and framework are chosen. After that, each module may use the test layout expected by its ecosystem, but tests must still trace back to a scenario or acceptance criterion.

Test categories:

- Contract tests: validate `TaskSpec`, `CommandSpec`, `AdapterMapping`, `ModelProfile`, `EvidenceSchema`, and `SessionEvent`.
- Adapter conformance tests: verify each CLI adapter satisfies the shared adapter contract.
- Policy tests: verify allow, deny, and approval decisions.
- Verification tests: prove verifier decisions use structured evidence.
- Replay tests: prove stored artifacts can be replayed by the eval plugin.

## Red Green Refactor

Red:

- Write the smallest failing test for one behavior.
- Confirm the failure comes from missing behavior, not a broken test.

Green:

- Implement the smallest change that satisfies the test.
- Avoid broad abstractions until duplication or contract pressure is visible.

Refactor:

- Simplify names, boundaries, and duplication.
- Keep tests passing after each refactor.

## Module Start Checklist

Before coding a module:

- Add or update one behavior scenario.
- Identify the contract object touched by the module.
- Define the expected evidence.
- Add a failing test for the first behavior.

Before finishing a module:

- Run the module test set.
- Run cross-module contract tests if a shared schema changed.
- Update docs when a contract or workflow changes.
- Store evidence in task artifacts.

## CLI Agent Rule

Coding agents must not claim a module is complete from narrative output alone. Completion requires verifier-readable evidence: passing tests, relevant logs, changed files, and known risks.
